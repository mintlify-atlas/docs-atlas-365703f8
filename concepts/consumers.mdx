## High-level vs. simple consumer

librdkafka provides two consumer interfaces:

<Tabs>
  <Tab title="High-level KafkaConsumer">
    The recommended interface for most applications. It manages consumer group membership, partition assignment, rebalancing, and offset commits automatically. Create it with `rd_kafka_new(RD_KAFKA_CONSUMER, ...)` and use `rd_kafka_subscribe()` or `rd_kafka_assign()`.
  </Tab>
  <Tab title="Simple (legacy) consumer">
    A lower-level interface where the application manually starts consumption per partition using `rd_kafka_consume_start()`. It does not participate in consumer groups. Prefer the high-level API for new applications.
  </Tab>
</Tabs>

## Creating a consumer

```c
rd_kafka_t *rk;
rd_kafka_conf_t *conf;
char errstr[512];

conf = rd_kafka_conf_new();
rd_kafka_conf_set(conf, "bootstrap.servers", "broker1:9092", errstr, sizeof(errstr));
rd_kafka_conf_set(conf, "group.id",          "my-group",     errstr, sizeof(errstr));
rd_kafka_conf_set(conf, "auto.offset.reset", "earliest",     errstr, sizeof(errstr));

/* rd_kafka_new() takes ownership of conf on success */
rk = rd_kafka_new(RD_KAFKA_CONSUMER, conf, errstr, sizeof(errstr));
if (!rk) {
    fprintf(stderr, "Failed to create consumer: %s\n", errstr);
    exit(1);
}

/* Redirect per-partition queues to the main queue for unified polling */
rd_kafka_poll_set_consumer(rk);
```

## Consumer groups

All consumers sharing the same `group.id` form a consumer group. The Kafka broker coordinates the group and distributes topic partitions among the members using the configured `partition.assignment.strategy` (default: `range,roundrobin`).

When a member joins or leaves the group, the broker triggers a **rebalance** to redistribute partitions. During a rebalance, consumption is briefly paused while partitions are reassigned.

<Note>
Set `group.id` before calling `rd_kafka_new()`. Without it, `rd_kafka_subscribe()` will fail.
</Note>

## Subscribe vs. assign

<Tabs>
  <Tab title="rd_kafka_subscribe()">
    Registers the consumer with the group coordinator for dynamic partition assignment. The broker assigns partitions and handles rebalancing automatically.

    ```c
    rd_kafka_topic_partition_list_t *topics =
        rd_kafka_topic_partition_list_new(2);
    rd_kafka_topic_partition_list_add(topics, "orders",   RD_KAFKA_PARTITION_UA);
    rd_kafka_topic_partition_list_add(topics, "invoices", RD_KAFKA_PARTITION_UA);

    rd_kafka_resp_err_t err = rd_kafka_subscribe(rk, topics);
    if (err) {
        fprintf(stderr, "Subscribe failed: %s\n", rd_kafka_err2str(err));
    }

    rd_kafka_topic_partition_list_destroy(topics);
    ```
  </Tab>
  <Tab title="rd_kafka_assign()">
    Manually pins the consumer to specific topic-partition-offset combinations. The consumer does not join a group and no rebalancing occurs. Use this when you need precise control over which partitions are consumed.

    ```c
    rd_kafka_topic_partition_list_t *parts =
        rd_kafka_topic_partition_list_new(1);
    rd_kafka_topic_partition_t *tp =
        rd_kafka_topic_partition_list_add(parts, "orders", 0);
    tp->offset = RD_KAFKA_OFFSET_BEGINNING;

    rd_kafka_assign(rk, parts);
    rd_kafka_topic_partition_list_destroy(parts);
    ```
  </Tab>
</Tabs>

## Poll loop

`rd_kafka_consumer_poll()` blocks until a message arrives or the timeout elapses. It also serves group heartbeats, rebalance callbacks, and other internal events.

```c
while (run) {
    rd_kafka_message_t *rkm = rd_kafka_consumer_poll(rk, 100 /* ms */);
    if (!rkm)
        continue; /* timeout, try again */

    if (rkm->err) {
        /* Informational consumer error — the client recovers automatically */
        fprintf(stderr, "Consumer error: %s\n",
                rd_kafka_message_errstr(rkm));
        rd_kafka_message_destroy(rkm);
        continue;
    }

    /* Process the message */
    printf("Received message: topic=%s partition=%" PRId32
           " offset=%" PRId64 " len=%zu\n",
           rd_kafka_topic_name(rkm->rkt),
           rkm->partition, rkm->offset, rkm->len);

    rd_kafka_message_destroy(rkm);
}
```

<Warning>
Call `rd_kafka_message_destroy()` on every message returned by `rd_kafka_consumer_poll()`, including error messages. Failing to do so leaks memory.
</Warning>

## Offset management

Offsets record the position of the last consumed message per topic+partition and are used to resume consumption after a restart or rebalance.

### Auto-commit

When `enable.auto.commit=true` (the default), librdkafka automatically commits the offset of the last message returned to the application every `auto.commit.interval.ms` milliseconds.

<Note>
Auto-commit runs in a background thread. If your application crashes after the commit but before finishing processing, the message is considered consumed and will not be redelivered. For at-least-once semantics, see below.
</Note>

### Manual commit

Disable automatic offset storage with `enable.auto.offset.store=false` and store offsets explicitly after processing. Auto-commit will then commit only offsets you have explicitly stored.

```c
/* After successfully processing a message */
rd_kafka_offsets_store(rk, rkm->rkt, rkm->partition,
                       rkm->offset + 1);
```

To commit immediately (synchronously):

```c
rd_kafka_commit_message(rk, rkm, 0 /* sync */);
```

### At-least-once processing

To guarantee that every message is processed at least once even across restarts:

1. Set `enable.auto.commit=true` and `enable.auto.offset.store=false`.
2. Process the message fully.
3. Call `rd_kafka_offsets_store()` only after successful processing.

```c
/* Process message first, store offset only on success */
if (process_message(rkm) == SUCCESS) {
    rd_kafka_offsets_store(rk, rkm->rkt, rkm->partition,
                           rkm->offset + 1);
}
rd_kafka_message_destroy(rkm);
```

<Tip>
Only offsets greater than the last committed offset are stored. Storing a lower offset has no effect.
</Tip>

## Auto offset reset

When there is no committed offset for a partition — for example, when a consumer group is first created — `auto.offset.reset` determines where to start reading:

| Value | Behaviour |
|-------|-----------|
| `earliest` | Start from the oldest available message in the partition |
| `latest` | Start from the next message to be produced (skip existing messages) |
| `error` | Raise `RD_KAFKA_RESP_ERR__AUTO_OFFSET_RESET` — application decides what to do |

```c
rd_kafka_conf_set(conf, "auto.offset.reset", "earliest", errstr, sizeof(errstr));
```

## Static consumer groups

By default, leaving and rejoining a consumer group triggers a rebalance. With **static group membership**, a consumer retains its partition assignment for up to `session.timeout.ms` after disconnecting, allowing brief restarts without disrupting other members.

To enable it, assign a unique `group.instance.id` to each consumer instance:

```c
rd_kafka_conf_set(conf, "group.instance.id", "consumer-host-1", errstr, sizeof(errstr));
```

Consumers with a `group.instance.id` do not send a leave-group request on close. The partition assignment is held until the session timeout expires or the same `group.instance.id` reconnects.

<Warning>
If two consumer instances start with the same `group.instance.id`, the existing member is fenced with a fatal error (`RD_KAFKA_RESP_ERR__FATAL`). Ensure each instance has a unique ID.
</Warning>

## Next-generation consumer group protocol (KIP-848)

librdkafka v2.12.0 and later support the next-generation consumer group protocol defined in KIP-848. It replaces the client-side group leader role with broker-side partition assignment distributed via heartbeats.

**Requirements:**

- Broker version v4.0.0 or later
- librdkafka v2.12.0 or later

**Enabling KIP-848:**

```c
rd_kafka_conf_set(conf, "group.protocol",      "consumer", errstr, sizeof(errstr));
rd_kafka_conf_set(conf, "group.remote.assignor", "uniform", errstr, sizeof(errstr));
/* group.remote.assignor is optional; leave unset to let the broker choose */
```

**Key differences from the classic protocol:**

| Aspect | Classic protocol | KIP-848 (consumer protocol) |
|--------|-----------------|------------------------------|
| Partition assignment | Client-side group leader | Broker-side coordinator |
| Rebalance type | Eager (stop-the-world) or cooperative-sticky | Incremental only |
| `session.timeout.ms` | Client configuration | Broker configuration (`group.consumer.session.timeout.ms`) |
| `heartbeat.interval.ms` | Client configuration | Broker configuration |
| Rebalance callback | `rd_kafka_assign()` / `rd_kafka_unassign()` | `rd_kafka_incremental_assign()` / `rd_kafka_incremental_unassign()` |

<Note>
When using `group.protocol=consumer`, the rebalance callback must use `rd_kafka_incremental_assign()` and `rd_kafka_incremental_unassign()`. Do not call `rd_kafka_assign()` inside the rebalance callback.
</Note>

## Proper termination sequence

```c
/* 1. Leave the consumer group and commit final offsets */
rd_kafka_consumer_close(rk);

/* 2. Destroy the consumer handle */
rd_kafka_destroy(rk);
```

<Note>
There is no need to call `rd_kafka_unsubscribe()` before `rd_kafka_consumer_close()`. If you omit `rd_kafka_consumer_close()`, final offsets are not committed and the consumer is kicked from the group only after `session.timeout.ms` expires.
</Note>

All objects obtained from the consumer (messages, topic-partition lists, events) must be destroyed before calling `rd_kafka_consumer_close()` and `rd_kafka_destroy()`.
