## Configuration API

librdkafka configuration is applied through an `rd_kafka_conf_t` object before the client handle is created. The same configuration API is used for both producers and consumers.

```c
rd_kafka_conf_t *conf;
rd_kafka_conf_res_t res;
char errstr[512];

/* Create a new configuration object */
conf = rd_kafka_conf_new();

/* Set a property */
res = rd_kafka_conf_set(conf, "bootstrap.servers", "broker1:9092,broker2:9092",
                        errstr, sizeof(errstr));
if (res != RD_KAFKA_CONF_OK) {
    fprintf(stderr, "Configuration error: %s\n", errstr);
    rd_kafka_conf_destroy(conf);
    exit(1);
}

/* Pass conf to rd_kafka_new() — it takes ownership on success */
rd_kafka_t *rk = rd_kafka_new(RD_KAFKA_PRODUCER, conf, errstr, sizeof(errstr));
if (!rk) {
    /* conf was NOT consumed; destroy it */
    rd_kafka_conf_destroy(conf);
    exit(1);
}
/* Do NOT reference conf after a successful rd_kafka_new() call */
```

`rd_kafka_conf_set()` returns one of:

| Result code | Meaning |
|-------------|---------|
| `RD_KAFKA_CONF_OK` | Property set successfully |
| `RD_KAFKA_CONF_UNKNOWN` | Unknown property name |
| `RD_KAFKA_CONF_INVALID` | Invalid value for the property |

<Warning>
`rd_kafka_conf_t` objects are not reusable after they have been passed to `rd_kafka_new()`. The library takes ownership and frees the object. If you need to create multiple clients, create a separate `rd_kafka_conf_t` for each one.
</Warning>

## Topic configuration

Per-topic settings are applied through `rd_kafka_topic_conf_t`. Create a topic object with `rd_kafka_topic_new()` after the client handle exists.

```c
rd_kafka_topic_conf_t *tconf = rd_kafka_topic_conf_new();

rd_kafka_topic_conf_set(tconf, "request.required.acks", "all",
                        errstr, sizeof(errstr));
rd_kafka_topic_conf_set(tconf, "message.timeout.ms", "30000",
                        errstr, sizeof(errstr));

/* tconf ownership is transferred to the topic object */
rd_kafka_topic_t *rkt = rd_kafka_topic_new(rk, "my-topic", tconf);
```

## Important properties

### Global properties (all client types)

| Property | Default | Description |
|----------|---------|-------------|
| `bootstrap.servers` | *(required)* | Comma-separated list of `host:port` broker addresses used for initial cluster discovery |
| `client.id` | `rdkafka` | Human-readable name for this client, visible in broker logs and metrics |
| `security.protocol` | `plaintext` | Transport protocol: `plaintext`, `ssl`, `sasl_plaintext`, or `sasl_ssl` |
| `group.id` | *(none)* | Consumer group identifier (required for consumers) |

### Producer properties

| Property | Default | Description |
|----------|---------|-------------|
| `linger.ms` | `5` | Time to wait for a batch to accumulate before sending |
| `batch.num.messages` | `10000` | Maximum messages per batch |
| `batch.size` | `1000000` | Maximum batch size in bytes |
| `request.required.acks` | `-1` (all) | Number of broker acknowledgements required: `0` = none, `1` = leader only, `-1` / `all` = all in-sync replicas |
| `message.timeout.ms` | `300000` | Maximum time librdkafka will attempt to deliver a message before failing it |
| `compression.codec` | `none` | Message compression: `none`, `gzip`, `snappy`, `lz4`, `zstd` |
| `enable.idempotence` | `false` | Enable exactly-once delivery and strict ordering |

### Consumer properties

| Property | Default | Description |
|----------|---------|-------------|
| `enable.auto.commit` | `true` | Automatically commit offsets in the background |
| `auto.commit.interval.ms` | `5000` | Frequency of background offset commits |
| `enable.auto.offset.store` | `true` | Automatically store the offset of each polled message |
| `fetch.min.bytes` | `1` | Minimum data the broker should return per FetchRequest |
| `fetch.wait.max.ms` | `500` | Maximum time the broker waits before responding to a FetchRequest |
| `session.timeout.ms` | `45000` | Consumer group session timeout; the broker removes the consumer if no heartbeat is received within this window |
| `heartbeat.interval.ms` | `3000` | Interval between consumer group heartbeats |
| `auto.offset.reset` | `largest` | What to do when there is no committed offset: `earliest`, `latest`, or `error` |

## Full configuration example

The following example mirrors the configuration snippet from the librdkafka introduction manual:

```c
rd_kafka_conf_t *conf;
rd_kafka_conf_res_t res;
rd_kafka_t *rk;
char errstr[512];

conf = rd_kafka_conf_new();

res = rd_kafka_conf_set(conf, "bootstrap.servers", "broker1:9092",
                        errstr, sizeof(errstr));
if (res != RD_KAFKA_CONF_OK) {
    fprintf(stderr, "%s\n", errstr);
    exit(1);
}

res = rd_kafka_conf_set(conf, "compression.codec", "snappy",
                        errstr, sizeof(errstr));
if (res != RD_KAFKA_CONF_OK) {
    fprintf(stderr, "%s\n", errstr);
    exit(1);
}

res = rd_kafka_conf_set(conf, "batch.num.messages", "100",
                        errstr, sizeof(errstr));
if (res != RD_KAFKA_CONF_OK) {
    fprintf(stderr, "%s\n", errstr);
    exit(1);
}

rk = rd_kafka_new(RD_KAFKA_PRODUCER, conf, errstr, sizeof(errstr));
if (!rk) {
    /* conf was NOT consumed; destroy it manually */
    rd_kafka_conf_destroy(conf);
    fprintf(stderr, "Failed to create producer: %s\n", errstr);
    exit(1);
}
/* conf is now owned by rk; do not use or free it */
```

Properties may be set in any order and may be overwritten before being passed to `rd_kafka_new()`. librdkafka validates the entire configuration at `rd_kafka_new()` time and will fail with a descriptive error if properties are incompatible (for example, `acks=1` combined with `enable.idempotence=true`).

## C++ configuration

The C++ interface uses `RdKafka::Conf` instead of `rd_kafka_conf_t`.

```cpp
#include <librdkafka/rdkafkacpp.h>

std::string errstr;

RdKafka::Conf *conf = RdKafka::Conf::create(RdKafka::Conf::CONF_GLOBAL);

if (conf->set("bootstrap.servers", "broker1:9092", errstr)
        != RdKafka::Conf::CONF_OK) {
    std::cerr << "Configuration error: " << errstr << std::endl;
    delete conf;
    exit(1);
}

if (conf->set("group.id", "my-consumer-group", errstr)
        != RdKafka::Conf::CONF_OK) {
    std::cerr << "Configuration error: " << errstr << std::endl;
    delete conf;
    exit(1);
}

/* Pass conf to the client constructor — it takes ownership */
RdKafka::KafkaConsumer *consumer =
    RdKafka::KafkaConsumer::create(conf, errstr);
if (!consumer) {
    std::cerr << "Failed to create consumer: " << errstr << std::endl;
    exit(1);
}
```

For per-topic configuration in C++, create a `CONF_TOPIC` object:

```cpp
RdKafka::Conf *tconf = RdKafka::Conf::create(RdKafka::Conf::CONF_TOPIC);
tconf->set("request.required.acks", "all", errstr);

/* Associate the topic conf with the global conf */
conf->set("default_topic_conf", tconf, errstr);
delete tconf;
```

<Note>
Like the C API, `RdKafka::Conf` objects must not be used after they have been passed to a client constructor. The constructor takes ownership of the configuration.
</Note>

## Callbacks via configuration

Callbacks are registered on the `rd_kafka_conf_t` object before creating the client, not on the handle itself. The following are the most commonly used callback setters:

```c
/* Delivery report callback (producer) */
rd_kafka_conf_set_dr_msg_cb(conf, dr_msg_cb);

/* Error callback (all client types) */
rd_kafka_conf_set_error_cb(conf, error_cb);

/* Rebalance callback (consumer) */
rd_kafka_conf_set_rebalance_cb(conf, rebalance_cb);

/* Statistics callback (all client types) */
rd_kafka_conf_set_stats_cb(conf, stats_cb);
```
