librdkafka supports all released Apache Kafka broker versions since **0.8.0**, but not all features are available on all broker versions — some features depend on newer broker functionality.

## Version configuration

librdkafka automatically negotiates features with the broker using the ApiVersions request (KIP-35). The relevant configuration properties are:

| Property | Default | Description |
|----------|---------|-------------|
| `api.version.request` | `true` | Request the broker's supported API versions to enable feature auto-negotiation. Requires broker >= 0.10.0. |
| `broker.version.fallback` | `0.10.0` | Broker version to assume when `api.version.request` fails or is disabled. Valid values: `0.8.0`, `0.8.1`, `0.8.2`, `0.9.0`. Any value >= 0.10 enables `api.version.request`. |
| `api.version.fallback.ms` | `0` | How long to use `broker.version.fallback` after an `api.version.request` failure before retrying. `0` = never revert. |

### Broker >= 0.10.0.0 (recommended)

For librdkafka >= v1.0.0 no version-related configuration is required — the defaults are set for broker 0.10.0.0 or later and feature negotiation is automatic.

```ini
# No configuration needed for >= 0.10.0.0 with librdkafka >= 1.0.0
bootstrap.servers=broker:9092
```

For librdkafka < v1.0.0, explicitly set:

```ini
api.version.request=true
api.version.fallback.ms=0
```

### Broker 0.9.0.x

```ini
api.version.request=false
broker.version.fallback=0.9.0.1
```

<Warning>
  Broker versions 0.9.0.0 and 0.9.0.1 contain a bug where the `ApiVersionRequest` will not close the connection but will time out after 10 seconds. librdkafka detects this and falls back to `broker.version.fallback` automatically, but the initial connection will be slower.
</Warning>

### Broker 0.8.x.y

```ini
api.version.request=false
broker.version.fallback=0.8.2.1
```

<Note>
  Broker versions 0.8.x are legacy and have very limited feature support. High-level consumer group management, SASL, SSL, and most Admin API operations are unavailable.
</Note>
